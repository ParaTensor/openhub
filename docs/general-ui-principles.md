# 通用的 UI/UX 设计与交互规范 (General UI Principles)

从我们在产品迭代中的具体实践里，可以进一步向上提炼出如下**更具普适性、适用于绝大多数产品研发及 Web 应用界面的通用 UI 要求与设计哲学**：

## 1. 结构刚性与防抖动 (Structural Stability & Anti-Flickering)
UI 的基础体验建立在“稳”上。任何不可预期的跳跃、闪烁和位移都会严重破坏系统的专业感。
- **避免隐式高度突变**：浮动层（Modal、Dropdown、Drawer）必须对内容高度有合理的预估或强制的最小/固定高度，避免因异步数据加载或表单校验错误导致面板忽长忽短。
- **弃用粗暴的 DOM 穿透**：严禁通过手动干预底层级 DOM（如 `document.body.style.overflow`）来解决滚动冲突。应该使用生态标准库（如 headless UI / Radix 等）所提供的高阶组件来处理无障碍（a11y）与焦点/滚动锁定。
- **弹窗隔离 (Modal-First) 原则**：遇到包含 3 个及以上复杂字段、或联动逻辑较重的编辑任务时，不要在列表或表格内进行“内联编辑 (Inline-edit)”，统一采用弹窗或侧边抽屉隔离交互上下文，保证主干视觉流不变。

## 2. 链路极简与视觉降噪 (Workflow Minimization & Visual Denoising)
减少用户的“认知负荷”，最好的 UI 是不需要阅读说明书的 UI。
- **消灭中间态**：坚决避免“草稿 (Draft)”等开发视角的冗长状态流。交互应是直达式的流向（Edit -> Publish），除非是长文写作等极特殊的场景，否则强事务操作应做到“不确认不生效，一确认必生效”。
- **视图扁平化**：拒绝无意义的选项卡（Tabs）和折叠面板。如果表单项能够被合理地排布在同一个层级（如采用双列或多列布局横向排布相关选项），就绝对不要将核心内容隐藏到“高级设置”中去，避免增加用户的查找成本。
- **文案克制原则**：
  - **去解释化**：“讲人话”，不要在界面上暴露出技术底层的逻辑代号。所有表单标签必须干净利落（如用最通用的说法替代内部代号）。
  - **去括号化**：坚决避免在表单 Label 中加入大量 `( )` 括弧及备注文字；如果字段确实需要提示风险或背景，一律收纳进 `Tooltip`（问号悬浮窗）中。
  - **去伪饰化**：少用“热门”、“推荐”等狗皮膏药式的徽章 (Badges)，除非有客观的数据支撑，否则应保持界面的中立与克制。

## 3. 防呆与自动化体验 (Error-proofing & Automation)
交互设计应当“顺着人性走，推着用户走”，而不是作为被动的机器等着用户发号施令。
- **预判与默认选中**：用户进入任何表单或配置界面时，只要有合理的业务推断，系统必须提供一个“默认值”（哪怕是随机选取），绝对不可以抛出空白界面逼迫用户从零做选择。
- **即时交互与隐去按钮**：
  - 涉及到数据换算或折扣应用时，去掉多余的“计算/应用”按钮。
  - 大量拥抱原生的表单事件（如 `onChange`, `onBlur`），用户输入完成、移开视线的瞬间，系统应当立刻在后台自动完成推算甚至防抖动保存，做到“所填即所得”。
- **全局通知的有效停留**：成功或失败提示（Toast/Notification）涉及页面跳转或关键状态改变时，通知必须有足够长的屏幕停留时间（通常至少 3~5 秒），不要做稍纵即逝的视觉盲区。

## 4. 全局一致性 (Global Consistency)
代码可以分模块解耦，但向用户展示的面必须是一个整体。
- **从不硬编码文案**：所有的界面文本内容不能在业务组件中随手硬编码。无论是大标题还是错误的占位符，都必须通过统一的多语言环境库（如 i18n 字典）统一管理，强制维持任何语言下全站词汇的高度同一。
- **所见即真实**：UI 是对数据的直接投影。绝不在视图层中写死虚假的状态模拟数据（Mock），只有在请求到确切后端状态时，UI 才能被唤醒，进而避免由前端表现与后端逻辑脱节引发的安全隐患和信任危机。
